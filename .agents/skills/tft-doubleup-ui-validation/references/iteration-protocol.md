# 반복 검증 프로토콜

## 기본 원칙

검증 실패 후에는 실패 항목만 다시 보지 않는다. 수정한 뒤 항상 전체 체크리스트를 처음부터 다시 실행한다.

Figma 작업에서는 시각 검증 전에 산출물 구조를 먼저 검증한다. 필요한 페이지나 최종 프레임이 없거나 children 수가 0이면 스크린샷이 일부 좋아 보여도 즉시 `불합격`이다.

검증 효율성보다 누락 방지를 우선한다. 모든 요소를 하나씩 확인하지 못했다면 검증은 끝난 것이 아니다.

## 판정

- 모든 항목 통과: `통과`
- 하나라도 문제 있음: `불합격`
- 애매함: `불합격`

## 실패 유형

작은 수정:

- 버튼 padding 이상
- 특정 카드 정렬 불일치
- 태그 간격 불규칙
- 색상 대비 부족
- 텍스트 줄바꿈 문제
- 숫자 배지와 텍스트 그룹 간 gap 부족
- 작은 리스트 행에서 배지가 제목을 침범함
- 모집 카드 안의 특정 배지/태그/버튼이 다른 요소와 겹침
- 특정 카드의 작성자명/메타 텍스트가 버튼 또는 태그와 겹침
- 모바일 특정 카드의 배지가 오른쪽 경계를 넘음

부분 재제작:

- 모집 카드 구조가 반복에 부적합
- 필터 패널 정렬 체계가 불안정
- 헤더 균형이 무너짐
- 모바일 전환 구조가 없음
- 최근 모집글/내 신청 상태 패널 리스트가 좌표 기반이라 반복 행이 깨짐
- 모바일 카드 내부 배지/태그/액션 행이 화면 밖으로 밀림

전체 재제작:

- 화면이 랜딩페이지처럼 보이고 실제 정보가 늦게 나옴
- 좌표 기반 배치가 많아 반응형으로 옮기기 어려움
- Atomic Design 계층이 무너져 컴포넌트 재사용이 어려움
- lolchess.gg 톤과 크게 다름
- UI 전반이 서비스로 보기 어색함
- Figma 페이지 구조가 목표와 다르거나 최종 프레임이 누락됨
- Figma MCP 오류 이후 빈 페이지 또는 일부 프레임만 남은 상태로 산출물이 불안정함
- MVP에서 제외된 파트너 추천/랭킹/가이드가 실제 기능처럼 만들어짐

## 반복 절차

1. Figma 페이지 목록과 각 페이지 children 수를 확인한다.
2. 필요한 최종 프레임 id와 크기를 확인한다.
3. 전체 UI 캡처 또는 Figma 화면을 본다.
4. 각 최종 화면을 개별로 캡처한다.
5. 각 화면의 모든 섹션을 개별로 확인한다.
6. 각 섹션의 모든 반복 아이템을 하나씩 확인한다.
7. 각 반복 아이템의 모든 자식 요소를 하나씩 확인한다.
8. 우측 패널, 리스트 행, 모바일 카드 같은 작은 UI를 확대 또는 개별 노드 스크린샷으로 본다.
9. 체크리스트를 순서대로 검토한다.
10. 실패 항목을 기록한다.
11. 수정 범위를 정한다.
12. 수정한다.
13. 전체 검증을 처음부터 다시 한다.
14. 실패가 없어질 때까지 반복한다.

## 중단하지 말아야 할 상황

- 첫 번째 수정 후 일부만 나아진 상태
- 데스크톱은 괜찮지만 모바일이 미검증인 상태
- Figma는 보기 괜찮지만 부모-자식 구조가 불명확한 상태
- 버튼/태그/카드 padding이 감으로만 맞춰진 상태
- Figma 페이지 수와 최종 프레임 수를 확인하지 않은 상태
- 전체 화면 축소 스크린샷만 보고 작은 패널 리스트를 확대 검증하지 않은 상태
- 스크린샷에 깨진 UI가 보이는데도 작은 문제라고 판단한 상태
- 모집 카드 중 일부만 확인한 상태
- 우측 패널 리스트 행 중 일부만 확인한 상태
- 모바일 카드 중 일부만 확인한 상태
- 같은 컴포넌트라며 실제 인스턴스 전수 검증을 생략한 상태
- 화면이 작아 판단이 어려운데 확대 캡처 없이 통과 처리한 상태

이런 경우에는 완료하지 말고 계속 수정과 전체 재검증을 반복한다.
