# TFT 더블업 UI 검증

## 목적

이 스킬은 Figma 또는 구현된 UI가 실제 서비스로 보기 어색하지 않은지 검증하고, 실패하면 다시 제작 또는 수정한 뒤 처음부터 전체 검증을 반복하기 위한 기준이다. 목표는 “대충 보기 괜찮음”이 아니라, 명확한 UI 품질 기준을 통과하는 것이다.

## 사용 관계

- UI를 새로 만들거나 Figma에 작성할 때는 먼저 `tft-doubleup-figma-design`를 따른다.
- 제품 방향, 화면 목적, 문구가 흔들리면 `tft-doubleup-product`를 따른다.
- 화면을 만든 뒤에는 이 스킬로 검증한다.
- 검증 실패 시 원인을 고친 뒤, 실패 항목만 다시 보는 것이 아니라 전체 검증을 처음부터 다시 한다.

## 검증 루프

반드시 아래 순서를 반복한다.

1. 현재 UI 산출물을 확보한다.
   - Figma 화면이면 스크린샷 또는 Figma 노드 상태를 확인한다.
   - 구현된 웹 UI면 브라우저 스크린샷과 DOM/CSS 상태를 확인한다.
   - Figma 파일이면 먼저 페이지 목록과 각 페이지의 최상위 children 수를 확인한다.
   - 사용자가 요구한 페이지/프레임이 없거나 비어 있으면 스크린샷 검증 전에 `불합격`으로 판정한다.

2. 전체 체크리스트를 처음부터 끝까지 검증한다.
   - Figma 페이지/프레임 존재 여부
   - 레이아웃 구조
   - 정렬
   - spacing/padding
   - 타이포그래피
   - 버튼/입력/카드 상태
   - 반응형
   - lolchess.gg 톤앤매너
   - Atomic Design 계층
   - 부모-자식 flexbox형 구조
   - 작은 컴포넌트와 우측 패널 리스트의 텍스트/배지 겹침 여부
   - 왼쪽 필터 패널의 필터 그룹/옵션 칩 overflow 여부
   - 주요 컨테이너의 자식 요소가 부모 `absoluteBoundingBox` 밖으로 나가지 않는지 여부
   - 반복 모집 카드의 내부 행 y offset, 액션 버튼 위치, 카드 간 리듬 일관성
   - 마지막 모집 카드와 viewport/부모 섹션 하단 사이의 최소 여백
   - 모든 화면의 모든 UI 요소 단위 검증

3. 하나라도 실패하면 `불합격`으로 판정한다.

4. 실패 원인을 기록한다.
   - 어떤 화면/프레임/컴포넌트인가
   - 무엇이 이상한가
   - 왜 일반적인 UI 기준에서 문제인가
   - 수정 방향은 무엇인가

5. 수정 또는 재제작한다.
   - 작은 문제면 해당 컴포넌트를 수정한다.
   - 구조적 문제면 해당 화면 또는 컴포넌트 계층을 다시 만든다.
   - UI가 전반적으로 어색하면 부분 수정에 집착하지 말고 화면을 새로 만든다.

6. 수정 후에는 다시 1번부터 전체 검증한다.

7. 모든 기준을 통과할 때만 완료한다.
   - 스크린샷에서 눈에 보이는 깨짐이 하나라도 있으면 완료하지 않는다.
   - `대충 보기에 큰 문제 없음`은 통과가 아니다.

## 합격 기준

모든 항목이 `통과`여야 한다. `대체로 괜찮음`, `나중에 수정 가능`, `눈에 크게 안 띔`은 합격이 아니다.

합격 조건:

- 필요한 Figma 페이지와 최종 프레임이 모두 존재하고 비어 있지 않다.
- 화면이 깨져 보이지 않는다.
- 정보 위계가 명확하다.
- padding, gap, align이 일반적인 UI 기준에서 자연스럽다.
- 텍스트가 잘리거나 겹치지 않는다.
- 버튼/입력/카드 크기가 용도에 맞다.
- 헤더와 주요 섹션이 한쪽으로 비정상적으로 쏠려 보이지 않는다.
- Atomic Design 계층이 무너지지 않는다.
- 주요 레이아웃이 좌표 기반이 아니라 부모-자식 flexbox형 구조다.
- lolchess.gg와 유사한 다크 정보형 TFT 도구 톤이 유지된다.
- 우측 패널, 최근 모집글 행, 내 신청 상태 행, 모바일 카드 같은 작은 UI에서도 배지와 텍스트가 서로 침범하지 않는다.
- 왼쪽 필터 패널의 모든 필터 그룹과 옵션 칩이 패널 경계 안에 있으며 중앙 모집 리스트를 침범하지 않는다.
- 주요 컨테이너의 자식 요소가 부모 영역 밖으로 나가지 않는다.
- 반복 모집 카드의 `Header`, `Title`, `Meta`, `Tags`, `Actions`, `신청하기` 버튼 위치가 카드마다 같은 상대 좌표 리듬을 유지한다.
- 마지막 모집 카드가 viewport 또는 부모 섹션 하단에 붙지 않고 충분한 하단 여백을 가진다.
- MVP에서 `파트너 추천`, `랭킹`, `가이드`가 실제 기능/패널로 노출되지 않고 `준비중입니다.` 상태로만 처리된다.
- 모든 UI 요소를 하나씩 검사했고, 검사하지 않은 영역이 없다.

## 실패 시 재작업 기준

다음 중 하나라도 있으면 수정 또는 재제작한다.

- 버튼 padding이 지나치게 작거나 커서 텍스트와 균형이 맞지 않는다.
- 버튼 높이, 입력 높이, 태그 높이가 서로 어색하게 다르다.
- 헤더 로고/네비/액션이 한쪽에 비정상적으로 몰려 있다.
- 카드 내부 정보가 위아래 또는 좌우로 불균형하게 쏠려 있다.
- 리스트 카드 간격이 불규칙하다.
- 필터 패널과 콘텐츠 리스트의 정렬 기준선이 맞지 않는다.
- 텍스트가 부모 영역을 넘거나 줄바꿈이 어색하다.
- 모바일에서 버튼이나 태그가 터치하기 어려울 정도로 작다.
- 다크 배경 대비가 부족해 읽기 어렵다.
- 장식이 정보보다 앞서거나, 정보형 서비스보다 랜딩페이지처럼 보인다.
- absolute/좌표 배치 때문에 반응형 구조가 깨질 위험이 있다.
- Figma 페이지가 누락됐거나 목표 페이지의 children 수가 0이다.
- 최종 화면 프레임이 누락됐거나 빈 프레임이다.
- 숫자 배지, 티어 배지, 상태 배지, 태그가 제목/메타 텍스트 위를 덮는다.
- 한국어 라벨이 의도 없이 두 줄로 쪼개지거나 버튼/배지 밖으로 넘친다.
- 우측 패널의 리스트에서 배지와 텍스트 그룹 간 gap이 8px 미만이거나 시각적으로 붙어 보인다.
- 필터 그룹 또는 필터 옵션 칩이 왼쪽 필터 패널 밖으로 삐져나오거나 중앙 모집 리스트 영역을 침범한다.
- Figma에서 부모 컨테이너의 `clipsContent`로 overflow를 숨겨서 통과처럼 보이게 만든다.
- 반복 모집 카드의 `신청하기` 버튼 right edge가 카드마다 2px 이상 다르다.
- 반복 모집 카드의 `신청하기` 버튼 top offset 또는 `Actions` row top offset이 카드마다 2px 이상 다르다.
- 같은 카드 컴포넌트처럼 보이는데 카드별 `Header`, `Title`, `Meta`, `Tags`, `Actions` 행의 상대 y 위치가 흔들린다.
- 카드 내부 padding을 줄여 viewport 안에 억지로 넣었지만 작성자 행, 태그, 버튼의 수직 정렬이 깨진다.
- 마지막 모집 카드 bottom과 viewport bottom 사이의 여백이 24px 미만이다.
- 마지막 모집 카드 bottom과 `Layout/ContentGrid` bottom 사이의 여백이 16px 미만이다.
- MVP 화면에 실제 파트너 추천 목록, 실제 랭킹 목록, 실제 가이드 콘텐츠가 노출된다.

## 검증 강도

검증은 관대하게 하지 않는다. 애매하면 실패로 보고 수정한다.

단, 모든 차이를 문제로 보지는 않는다. 아래는 허용한다.

- 의도된 정보 밀도
- lolchess.gg 스타일의 촘촘한 리스트
- 작은 배지와 태그
- 다크 UI 특유의 낮은 채도

허용하지 않는다.

- 의도 없이 좁거나 넓은 padding
- 기준선이 맞지 않는 헤더
- 카드마다 다른 정보 배치
- 반복 카드마다 신청 버튼 위치나 작성자/액션 행 기준선이 다른 상태
- 모집 리스트 마지막 카드가 화면 하단에 딱 붙은 상태
- 클릭 영역이 부족한 버튼
- 모바일에서 읽기 어려운 텍스트
- 부모-자식 구조 없이 좌표로만 배치된 주요 레이아웃
- 페이지/프레임 존재 검증 없이 스크린샷만 보고 통과 처리하는 것
- 스크린샷에 보이는 깨진 작은 컴포넌트를 무시하고 통과 처리하는 것
- Figma MCP 조회가 불안정한데도 추가 확인 없이 완료 보고하는 것

## 전수 검증 원칙

효율성보다 누락 방지를 우선한다. Figma UI 검증에서는 화면을 “전체적으로 괜찮아 보이는지” 보는 것으로 끝내지 않는다. 모든 화면, 모든 프레임, 모든 반복 항목, 모든 배지/태그/텍스트/버튼/입력 요소를 하나씩 검증한다.

반드시 지킨다.

- 최종 화면 1개씩 따로 검증한다.
- 화면 안의 섹션 1개씩 따로 검증한다.
- 섹션 안의 반복 아이템 1개씩 따로 검증한다.
- 반복 아이템 안의 모든 자식 요소를 하나씩 검증한다.
- 카드가 4개 있으면 4개 모두 검증한다. 첫 번째 카드만 보고 나머지를 추정하지 않는다.
- 같은 컴포넌트처럼 보여도 실제 인스턴스가 다르면 모두 확인한다.
- 같은 이름의 반복 컴포넌트는 개별 검증 후 서로 상대 좌표를 비교한다.
- 축소 스크린샷에서 작은 요소가 잘 안 보이면 해당 노드를 확대하거나 개별 스크린샷을 찍는다.
- 검증하지 못한 요소가 있으면 `미검증`이 아니라 `불합격`이다.

모든 요소에서 확인할 항목:

- 부모 영역 밖으로 넘치지 않는가
- 다른 텍스트, 배지, 버튼, 카드 경계와 겹치지 않는가
- 텍스트가 의도 없이 줄바꿈되지 않는가
- 텍스트가 잘리거나 말줄임 없이 필요한 정보를 읽을 수 있는가
- 배지/태그 안의 텍스트가 중앙 정렬되어 보이는가
- 배지/태그 내부 padding이 충분한가
- 인접 요소와 최소 8px 이상의 시각적 gap이 있는가
- 클릭 가능한 요소의 높이와 폭이 용도에 맞는가
- 다크 배경에서 대비가 충분한가
- 반복 항목 사이의 정렬 기준선이 일관적인가

모집 카드 전수 검증:

- 카드 컨테이너
- KR 서버 배지
- 티어 배지
- 가능 상태 배지
- 매칭 적합도 배지
- 제목 텍스트
- 목적 배지
- 디스코드 배지
- 모든 플레이 스타일 태그
- 작성자명
- 작성자 메타 텍스트
- 신청 버튼
- 카드 내부 행 간 간격
- 카드 간 간격
- 카드 top 기준 `Header`, `Title`, `Meta`, `Tags`, `Actions` 행의 y offset
- `신청하기` 버튼의 right edge와 카드 top 기준 y offset
- 작성자 프로필 행과 `신청하기` 버튼의 수직 중앙 정렬

MVP에서는 예약제와 미래 시간대 모집이 없으므로 모집 카드에 `평일 21:00`, `주말 오후`, `오늘 22:00`, `시간 조율` 같은 시간 배지가 있으면 `불합격`이다. 필요한 경우 `바로 1판`, `바로 2판`, `현재 플레이 중`처럼 즉시 가능 상태로 표현한다.
디스코드 필드 타이틀은 `디스코드`여야 하며 선택지는 `대화 가능`, `듣기만 가능`, `디스코드 불가능` 3개만 허용한다. `Discord 가능`, `음성 필수`, `음성 선택`, `음성 없이 가능`이 보이면 `불합격`이다.

우측/상세 패널 리스트 전수 검증:

- 패널 컨테이너
- 패널 제목
- 패널 우측 메타 라벨
- 모든 숫자 배지
- 모든 리스트 제목
- 모든 리스트 메타 텍스트
- 숫자 배지와 텍스트 그룹의 gap
- 행 간 간격

모바일 전수 검증:

- 상단 브랜드
- 상단 액션 버튼
- 검색 필드
- 필터 요약 배지 전체
- 모든 모바일 모집 카드
- 모바일 카드 안의 모든 배지/태그/텍스트/버튼
- 하단 액션바
- 하단 액션바가 viewport 안에 완전히 들어오는지
- 좌우 여백과 터치 영역

최종 보고 전에는 “전수 검증 완료”라고 말할 수 있으려면 위 항목을 실제로 모두 확인해야 한다. 하나라도 스크린샷에서 흐리거나 작아서 판단하지 못했으면 확대 검증 후 다시 판정한다.

## Figma 구조 검증

Figma UI를 검증할 때는 시각 검증 전에 구조 검증을 먼저 한다.

필수 확인:

- 파일에 필요한 페이지가 존재하는가
- 각 페이지 children 수가 0이 아닌가
- `05 Pages`에 최종 화면 프레임이 실제로 존재하는가
- 각 최종 화면 프레임 크기가 의도한 viewport와 맞는가
- 스크린샷을 찍을 node id가 실제 최종 프레임 id인가

기본 페이지 세트가 목표일 때는 아래 7개가 모두 있어야 한다.

- `00 Foundation`
- `01 Atoms`
- `02 Molecules`
- `03 Organisms`
- `04 Templates`
- `05 Pages`
- `99 Notes`

기본 최종 프레임 세트는 아래 3개다.

- `Page/Desktop Matching Board`
- `Page/Desktop Recruitment Detail`
- `Page/Mobile Matching Board`

하나라도 없거나 비어 있으면 `불합격`이다.

## 작은 UI 확대 검증

전체 화면 축소 스크린샷만으로 통과 처리하지 않는다. 다음 영역은 반드시 확대하거나 개별 노드 스크린샷으로 확인한다.

- 우측 패널의 `최근 모집글`
- 우측 패널의 `내 신청 상태`
- 우측 패널의 `서비스 공지`
- 상세 화면의 `비슷한 모집글`
- 모집 카드의 배지/태그/작성자/신청 버튼 행
- 왼쪽 필터 패널의 모든 필터 그룹과 옵션 칩
- 모바일 카드의 배지/태그/액션 행
- 모바일 하단 액션바

MVP에서는 `파트너 추천`, `랭킹`, `가이드` 화면/패널을 검증 대상으로 만들지 않는다. 해당 메뉴를 표현해야 한다면 `준비중입니다.` 모달 또는 상태만 검증한다.

검증 기준:

- 숫자 배지는 텍스트를 덮지 않는다.
- 배지와 텍스트 그룹 사이에는 최소 8px 이상의 시각적 gap이 있다.
- 한 줄 배지 라벨은 의도 없이 두 줄로 쪼개지지 않는다.
- 텍스트가 카드, 패널, 버튼, 배지 경계를 넘지 않는다.
- 필터 옵션 칩이 필터 패널 경계를 넘거나 중앙 리스트 위로 올라타지 않는다.
- 모바일에서 버튼과 태그가 화면 밖으로 밀리지 않는다.

## 왼쪽 필터 패널 전수 검증

데스크톱 메인 화면의 `Organism/Filters/Sidebar`는 우측 패널과 같은 강도로 확대 검증한다. 전체 화면 축소 스크린샷에서 대략 보인다는 이유로 통과 처리하지 않는다.

필수 확인:

- `Organism/Filters/Sidebar` 컨테이너
- 모든 `Molecule/FilterGroup/*`
- 각 필터 그룹의 제목
- 각 필터 그룹 안의 모든 옵션 칩
- `필터 적용`, `초기화` 버튼
- 필터 패널과 중앙 모집 리스트 사이의 경계

실패 조건:

- 필터 그룹 또는 옵션 칩이 `Organism/Filters/Sidebar`의 좌우 경계를 1px 이상 넘으면 `불합격`이다.
- 옵션 칩, 텍스트, 버튼이 중앙 모집 리스트 카드 위로 침범하면 `불합격`이다.
- 긴 한국어 옵션 칩이 부모 폭 안에서 wrap되지 않고 밖으로 나가면 `불합격`이다.
- 옵션 칩 row가 부모 폭보다 넓은 hug contents 상태로 남아 있으면 `불합격`이다.
- `clipsContent=true`로 삐져나온 요소를 숨긴 경우에도 구조적으로는 `불합격`이다. 부모 폭에 맞게 wrap, fixed width, fill container를 수정해야 한다.

## 부모 영역 overflow 기하 검증

Figma 최종 화면 검증 시 스크린샷 확인과 별도로 주요 컨테이너의 자식 `absoluteBoundingBox`를 검사한다. 보이는 화면에서 애매하면 도구로 좌표를 확인한다.

검사 대상:

- `Organism/Filters/Sidebar`
- 모든 `Molecule/FilterGroup/*`
- `Organism/RecruitmentCard/*`
- `Layout/RightRail`
- 모든 `Organism/Panel/*`
- `Page/Mobile Matching Board` 내부 카드와 하단 액션바

불합격 조건:

- 자식의 right edge가 부모 right edge를 1px 이상 넘는다.
- 자식의 left edge가 부모 left edge를 1px 이상 넘는다.
- 자식의 bottom edge가 부모 bottom edge를 1px 이상 넘고 의도된 스크롤/클리핑 구조가 아니다.
- 텍스트, 배지, 태그, 버튼이 부모 컨테이너 밖으로 나간다.
- 허용된 absolute overlay가 아닌데 부모 밖에 위치한다.

허용 예외:

- `준비중입니다.` 모달, 드롭다운, 알림 배지처럼 의도된 overlay이며 레이어 이름과 목적이 명확한 경우
- 최종 화면 viewport 밖으로 나가지 않고, 다른 정보 영역을 침범하지 않는 팝오버

## 반복 카드 리듬 검증

모집 카드가 여러 개 있으면 모든 카드를 서로 비교한다. 각 카드가 개별적으로 부모 밖으로 나가지 않는 것만으로는 통과가 아니다.

필수 확인:

- 모든 `Organism/RecruitmentCard/Default`의 width가 동일한가
- 모든 카드의 height가 동일하거나, 콘텐츠 차이로 달라진 경우에도 내부 섹션 순서와 기준선이 일관적인가
- 각 카드의 `Header`, `Title`, `Molecule/MetaRow/PostInfo`, `Tags`, `Actions` 행이 카드 top 기준으로 같은 y offset에 있는가
- 모든 `신청하기` 버튼의 right edge가 같은 x 좌표에 정렬되어 있는가
- 모든 `신청하기` 버튼의 카드 top 기준 y offset이 같은가
- 작성자 프로필 행과 `신청하기` 버튼이 같은 `Actions` row 안에서 수직 중앙 정렬되어 있는가
- 카드 내부 top/bottom padding이 카드마다 다르게 보이지 않는가
- 카드 간 gap이 일정한가

불합격 조건:

- 반복 카드의 `신청하기` 버튼 right edge 차이가 2px 이상이면 `불합격`이다.
- 반복 카드의 `신청하기` 버튼 top offset 차이가 2px 이상이면 `불합격`이다.
- 반복 카드의 `Actions` row top offset 차이가 2px 이상이면 `불합격`이다.
- 같은 카드 컴포넌트처럼 보이는데 카드별 내부 행 y offset이 다르면 `불합격`이다.
- 카드마다 작성자 행과 신청 버튼의 수직 정렬 기준선이 다르면 `불합격`이다.
- 카드 높이를 줄여 해결한 뒤 내부 row가 눌리거나 bottom padding이 사라지면 `불합격`이다.

## 리스트 하단 여백 검증

최종 화면에서 리스트 마지막 항목이 viewport 또는 부모 섹션 하단에 붙어 있으면 `불합격`이다.

필수 기준:

- `Page/Desktop Matching Board`에서 마지막 모집 카드 bottom과 viewport bottom 사이에 최소 24px 이상 여백이 있어야 한다.
- 마지막 모집 카드 bottom과 `Layout/ContentGrid` bottom 사이에 최소 16px 이상 여백이 있어야 한다.
- 카드가 많아 한 화면에 모두 넣기 어렵다면, 카드 높이를 무리하게 압축하지 말고 리스트 스크롤, 클리핑, 더보기 구조를 명확히 표현한다.
- 모든 카드를 억지로 viewport 안에 넣기 위해 카드 내부 padding이나 액션 행 정렬을 깨뜨리면 `불합격`이다.

불합격 조건:

- 마지막 카드가 viewport 하단에 24px 미만으로 붙어 보인다.
- 카드 리스트가 하단 safe area 없이 끝난다.
- 하단 여백 확보를 위해 카드 내부 요소가 눌리거나 버튼 위치가 흔들린다.

## 반복 컴포넌트 동일성 기하 검증

Figma QA에서 같은 이름의 반복 컴포넌트 인스턴스는 상대 좌표를 비교한다.

검사 대상:

- `Organism/RecruitmentCard/Default`
- `Organism/RecruitmentCard/Compact`
- `PanelRow`
- 필터 옵션 칩 row

검증 방식:

- 각 반복 컴포넌트의 parent-relative child bounding box를 수집한다.
- 같은 역할의 child node는 x, y, width, height 차이를 비교한다.
- 텍스트 길이 때문에 width가 달라질 수 있는 요소는 제외하되, 시작 x, top y, row height, 버튼 right edge는 고정 기준으로 본다.

불합격 조건:

- 같은 역할의 액션 버튼 right edge가 카드마다 다르다.
- 같은 역할의 행 top offset이 카드마다 다르다.
- 같은 역할의 row height가 카드마다 다르다.
- 반복 컴포넌트의 내부 spacing이 카드마다 다르게 보인다.
