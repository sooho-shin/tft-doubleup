# TFT 더블업 매칭 서비스

## 목적

이 스킬은 TFT 더블업 팀원 찾기 서비스를 일관되게 기획하고 디자인하기 위한 Codex 작업 지침이다. 실제 최신 제품 기획의 source of truth는 저장소의 `docs/` 문서이며, 이 스킬은 Codex가 그 문서를 어떤 기준으로 읽고 활용할지 정한다.

이 서비스는 일반 커뮤니티 게시판이 아니라, TFT 유저가 조건에 맞는 더블업 파트너를 빠르게 찾는 정보 밀도 높은 매칭 보드로 다룬다.

## 문서 우선순위

기획, 구현, 디자인 판단은 아래 문서를 우선한다.

- `docs/product/mvp.md`: MVP 범위, 포함/제외 기능, 핵심 사용자 흐름
- `docs/product/screens.md`: 화면 목록, 화면별 필수 상태, 데스크톱/모바일 레이아웃
- `docs/product/ux-copy.md`: 한국어 UX 문구, 라벨, 버튼, 금지 문구
- `docs/product/design-direction.md`: 시각 방향, 색상, 컴포넌트 기준
- `docs/domain.md`: 도메인 모델, 상태값, enum 후보
- `docs/harness.md`: 하네스 운영 방식

`docs/`와 이 스킬의 `references/` 내용이 충돌하면 `docs/`를 우선한다. `references/`는 배경 자료와 초기 판단 기준으로만 사용한다.

## 기본 작업 흐름

1. 지금 만들 산출물을 먼저 구분한다.
   - 제품 전략
   - MVP 범위
   - 사용자 흐름
   - 정보 구조
   - UX 문구
   - Figma 디자인
   - 구현 가능한 화면 명세

2. 모든 판단은 핵심 사용자 문제에 맞춘다.
   - 더블업을 같이 할 파트너를 빠르게 찾는다.
   - MVP에서는 한국 서버(KR) 안에서 지금 바로 플레이 가능한 사람을 중심으로 티어, 디스코드 대화 방식, 플레이 성향이 맞지 않아 생기는 실패 확률을 줄인다.

3. 작업 성격에 맞는 `docs/` 문서를 먼저 읽는다.
   - 범위와 우선순위: `docs/product/mvp.md`
   - 화면과 상태: `docs/product/screens.md`
   - 문구와 라벨: `docs/product/ux-copy.md`
   - 시각 방향: `docs/product/design-direction.md`
   - 데이터와 상태값: `docs/domain.md`

4. `docs/`에 없는 배경 설명이 필요할 때만 스킬 레퍼런스를 읽는다.
   - `references/product-brief.md`: 서비스 포지셔닝, 초기 MVP 배경
   - `references/design-direction.md`: lolchess.gg 느낌의 다크 정보형 UI 배경
   - `references/ux-copy-ko.md`: 초기 한국어 라벨과 마이크로카피

5. 결과물은 실용적으로 만든다.
   - 짧고 우선순위가 분명한 추천안
   - 구체적인 화면 섹션과 필드
   - 상태가 포함된 컴포넌트 목록
   - Figma로 바로 옮길 수 있는 레이아웃 명세
   - 사용자가 구현 단계로 넘어가겠다고 했을 때만 구현 세부사항

## 제품 원칙

- 첫 화면은 랜딩페이지가 아니라 실제 팀원 찾기 화면이어야 한다.
- 성격이나 자기소개보다 매칭 적합성을 먼저 보여준다.
- TFT 전적/메타 사이트처럼 정보 밀도는 높되 빠르게 읽히게 만든다.
- MVP에서는 서버를 한국 서버(KR)로 고정하고, 지금 가능 여부, 더블업 티어, 목적, 디스코드 대화 방식, 플레이 스타일을 핵심 정보로 둔다.
- 서버 선택/다중 서버 매칭은 MVP 이후 확장으로 다룬다.
- 예약제, 미래 시간대 모집, 시간 조율 기반 매칭은 MVP에서 제외한다.
- Riot, TFT, lolchess의 로고나 고유 UI를 직접 베끼지 않는다. 정보형 도구의 분위기만 참고한다.

## Figma 작업 기준

Figma 도구를 호출하기 전에는 먼저 화면 범위를 말로 확정한다.

기본 화면 세트:

- 데스크톱 메인 매칭 보드
- 모집글 상세 또는 프로필 패널
- 요청받은 경우 모바일 리스트 화면

Figma에서는 다음을 지킨다.

- 네이티브 프레임과 오토레이아웃을 사용한다.
- 레이어 이름은 의미 있게 짓는다.
- 배지, 태그, 버튼, 카드처럼 반복되는 요소는 컴포넌트화한다.
- lolchess.gg는 다크하고 촘촘한 정보형 레이아웃 참고용으로만 사용한다.

## 응답 방식

사용자가 한국어로 말하면 한국어로 답한다. 단, 코드명, 컴포넌트명, 파일명은 영어가 명확하면 영어를 유지한다. 답변은 제품 의사결정에 도움이 되도록 짧고 구체적으로 작성한다.
