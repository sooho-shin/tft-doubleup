# Figma 작업 체크리스트

## 만들기 전

- 화면 목적이 확정됐는가
- 새 파일인지 기존 파일 수정인지 확정됐는가
- 이미지 에셋이 필요한지, Figma 도형/벡터로 충분한지 판단했는가
- 제품 기준은 `tft-doubleup-product`와 충돌하지 않는가
- lolchess.gg 최신 화면을 확인했는가
- lolchess.gg와 매우 가까운 다크 정보형 TFT 도구 톤을 목표로 잡았는가
- lolchess.gg의 로고, 고유 에셋, 실제 데이터, 정확한 레이아웃 복제는 피하고 있는가

## 구조

- 페이지가 목적별로 나뉘었는가
- `00 Foundation`, `01 Atoms`, `02 Molecules`, `03 Organisms`, `04 Templates`, `05 Pages` 구조가 필요한 범위만큼 만들어졌는가
- 반복 요소가 Atomic Design 계층에 맞게 컴포넌트로 분리됐는가
- Atoms가 Molecules/Organisms에 의존하지 않는가
- Molecules가 Organisms/Templates/Pages에 의존하지 않는가
- Templates와 Pages가 구분되어 있는가
- 오토레이아웃이 적용됐는가
- 주요 화면이 부모-자식 관계가 명확한 flexbox형 구조인가
- 화면 전체, 콘텐츠 그리드, 리스트, 카드 내부가 Auto Layout의 방향/gap/padding으로 구성됐는가
- 좌표 기반 배치가 불필요하게 쓰이지 않았는가
- absolute/overlay 성격의 배치가 필요한 경우 기준 부모 프레임 안에 묶여 있는가
- 레이어 이름이 의미 있는가
- 프레임 크기가 명확한가

## 화면

- 첫 화면에서 모집글이 바로 보이는가
- 마케팅 히어로 대신 lolchess처럼 실제 정보 섹션이 먼저 보이는가
- 상단 서비스 바와 주요 네비게이션이 촘촘한 정보 사이트처럼 구성됐는가
- 필터가 실제 사용자 판단 기준과 맞는가
- MVP에서 서버 선택 드롭다운이나 다중 서버 필터를 만들지 않았는가
- MVP에서 예약제/미래 시간대/시간 조율 UI를 만들지 않았는가
- 디스코드 필드 타이틀이 `디스코드`이고 선택지가 `대화 가능`, `듣기만 가능`, `디스코드 불가능` 3개뿐인가
- 모집 카드에서 KR 서버, 티어, 지금 가능 상태, 목적, 디스코드, 태그가 보이는가
- 최근 모집글, 내 신청 상태, 서비스 공지 같은 MVP 보조 정보 패널이 있는가
- 파트너 추천, 랭킹, 가이드를 실제 기능/패널처럼 만들지 않았는가
- 파트너 추천, 랭킹, 가이드 클릭 상태가 `준비중입니다.`로 표현되는가
- 신청 버튼이 잘 보이는가
- 텍스트가 잘리지 않는가

## 시각 품질

- 다크 배경에서 대비가 충분한가
- lolchess.gg와 유사한 작은 배지, 얇은 보더, 촘촘한 카드/테이블 리듬이 있는가
- 카드 radius가 과하지 않은가
- 장식이 정보보다 앞서지 않는가
- 티어 색상이 과하게 면적으로 쓰이지 않았는가
- 카드 안에 카드처럼 보이는 구조가 없는가

## 이미지 에셋

- 비트맵 에셋이 정말 필요한 경우에만 `imagegen` 스킬을 사용했는가
- 단순 아이콘/배지/UI 장식은 Figma 도형 또는 벡터로 처리했는가
- 생성 에셋은 workspace 안에 저장했는가
- Figma에는 `upload_assets`로 업로드했는가
- `$CODEX_HOME/generated_images`에만 남은 파일을 최종 산출물로 참조하지 않는가
- 사용자가 특정 모델명을 요청한 경우, 현재 도구에서 모델 선택 가능 여부와 fallback 조건을 명확히 설명했는가

## 전달

- Figma 링크를 제공했는가
- 만든 페이지와 프레임을 요약했는가
- 사용한 이미지 에셋 경로와 생성 방식이 필요한 만큼 보고됐는가
- 미정 사항과 다음 작업을 분리해서 설명했는가
