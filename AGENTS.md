## 프로젝트 운영 규칙

이 저장소는 TFT 더블업 팀원 매칭 서비스를 만들기 위한 작업 공간이다. Codex는 변경 전에 현재 파일 구조와 기존 `.agents/skills/` 지침을 확인하고, 사용자가 만든 미추적 파일이나 수정사항을 되돌리지 않는다.

## 기본 원칙

- 사용자가 명시적으로 계획만 요청하지 않는 한, 필요한 파일 수정과 검증까지 수행한다.
- 코드, 문서, 스크립트 변경은 작고 추적 가능한 단위로 유지한다.
- 프로젝트 전용 문서와 스킬 본문은 기본적으로 한글을 사용한다.
- YAML frontmatter의 `name`은 영어 kebab-case를 유지한다.
- 기술스택이 아직 확정되지 않은 영역은 임의로 크게 도입하지 말고, 기존 구조와 사용자의 현재 목표에 맞춰 보수적으로 선택한다.

## 기획 문서 우선순위

- 제품 기획의 source of truth는 `docs/` 아래 문서다.
- `.agents/skills/`는 Codex의 작업 방식과 판단 기준을 정하는 지침으로 사용한다.
- `docs/`와 `.agents/skills/`의 내용이 충돌하면 `docs/`를 우선한다.
- MVP 범위나 기능 우선순위는 `docs/product/mvp.md`를 기준으로 한다.
- 화면 구성은 `docs/product/screens.md`를 기준으로 한다.
- UX 문구와 라벨은 `docs/product/ux-copy.md`를 기준으로 한다.
- 디자인 방향은 `docs/product/design-direction.md`를 기준으로 한다.
- 도메인 모델과 상태값은 `docs/domain.md`를 기준으로 한다.
- 스킬의 제품 레퍼런스는 오래된 배경 자료일 수 있으므로, 구현이나 기획 변경 전에는 관련 `docs/` 문서를 먼저 확인한다.

## 스킬 사용

- 기획 수정, Figma 디자인 반영, UI 검증 반복을 한 번에 수행할 때는 `tft-doubleup-design-workflow` 스킬을 사용한다.
- 제품 기획, 사용자 흐름, UX 문구를 다룰 때는 `tft-doubleup-product` 스킬을 우선 확인한다.
- Figma 디자인 파일, 화면, 컴포넌트, 디자인 토큰을 만들거나 수정할 때는 `tft-doubleup-figma-design` 스킬을 사용한다.
- 구현 UI나 Figma UI의 레이아웃, 타이포, 반응형, 시각적 깨짐을 검증할 때는 `tft-doubleup-ui-validation` 스킬을 사용한다.

## 검증 규칙

- 코드나 설정을 변경한 뒤에는 가능한 경우 `./scripts/check.sh`를 실행한다.
- UI 변경이 포함되면 가능한 경우 `./scripts/validate-ui.sh`도 실행한다.
- 릴리즈나 배포 전 점검은 `./scripts/qa.sh`를 기준으로 한다.
- 검증 명령이 프로젝트 초기 상태 때문에 건너뛰어진 경우, 어떤 항목이 건너뛰어졌는지 사용자에게 보고한다.

## 스크립트 계약

- `scripts/` 아래 파일은 Codex와 사람이 함께 쓰는 안정적인 작업 버튼이다.
- 스크립트는 실패 시 즉시 중단하도록 `set -euo pipefail`을 사용한다.
- 스크립트는 도구가 아직 설치되지 않은 초기 상태에서도 명확한 메시지를 남기고 종료해야 한다.
- 패키지 매니저가 필요한 경우 기본값은 `pnpm`으로 둔다.

## 프론트엔드 방향

- 첫 화면은 마케팅 랜딩보다 실제 매칭 서비스의 사용 흐름을 우선한다.
- SaaS/운영툴 성격에 맞게 조용하고 정보 밀도가 높은 UI를 지향한다.
- 반복 사용자가 빠르게 스캔하고 비교할 수 있도록 명확한 상태, 필터, 정렬, 입력 흐름을 제공한다.
- UI 구현 후에는 데스크톱과 모바일에서 텍스트 겹침, 레이아웃 깨짐, 과도한 여백, 반응형 문제를 확인한다.

## 백엔드 방향

- 매칭, 사용자 프로필, 게임 티어/포지션/가능 시간 같은 핵심 도메인 모델은 명시적인 타입과 검증 스키마를 둔다.
- 외부 API나 크롤링이 필요해지면 호출부를 격리하고 실패/지연/레이트리밋을 다룬다.
- 데이터 저장소가 정해지기 전에는 영속성 추상화를 과하게 만들지 않는다.
